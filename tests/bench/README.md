# Thumbro benchmark harness

Measures **file size**, **perceptual quality** (SSIMULACRA2), and **performance**
(wall time + peak RSS) for Thumbro's handlers against the MediaWiki baselines
(**ImageMagick** as the binding pass/fail baseline; **GD** as a never-regress floor),
and applies the acceptance gate described under **Acceptance gate** below.

## Dev dependencies (Debian 12 dev image)

```sh
apt-get install -y imagemagick libvips-tools webp time
bash tests/bench/bin/install-ssimulacra2.sh   # SSIMULACRA2 metric (needs root + network)
```

- **SSIMULACRA2** (quality metric, v2 0–100): installed by `install-ssimulacra2.sh` as
  `ssimulacra2_rs`. NOTE: `libjxl-tools` 0.7 ships only SSIMULACRA **v1**, not v2, and
  no prebuilt v2 binary was available — so we install the pure-Python `ssimulacra2`
  PyPI package (pinned). It is correct but slow (~0.6 s/frame); swappable for a
  compiled binary later by editing `Ssimulacra2::$bin` + the install script.
- `convert` (imagemagick), `vips`/`vipsthumbnail` (libvips-tools): thumbnailing,
  reference downscale, and animated-frame extraction (`anim_dump` is NOT packaged, so
  animated-WebP frames are extracted with `convert -coalesce`).
- `time` (GNU time): peak-RSS capture.

## Run

```sh
php tests/bench/benchmark.php --out=/tmp/bench               # whole corpus
php tests/bench/benchmark.php --mime=image/jpeg --out=/tmp/bench
php tests/bench/benchmark.php --mime=image/gif --visual      # one MIME + contact sheet
```

Outputs `results.json`, a printed table, and (with `--visual`) `visual/index.html`.
Animation runs are slow (metric is per-frame ~0.6 s).

## Interpreting scores (read this)

- SSIMULACRA2 bands: ≥90 visually lossless, ≥70 high, ≥50 medium, <50 low.
- **Corpus must be structured content, never noise** — pure noise (e.g. plasma) scores
  terribly under a reference metric and is meaningless.
  (Exception: `tiny.png` is a perf/size-extreme fixture at near-zero downscale, where content doesn't affect the score.)
- **Very small thumbnails (≤120px) give unstable, low SSIMULACRA2** regardless of real
  quality (the metric has too few scales). Treat ≤120px scores with low confidence and
  corroborate via the `--visual` contact sheet. (The representative tier measures at
  180/250/400; this caveat mainly concerns the small stress fixtures.)
- **Transparent content** is flattened over a fixed grey background before scoring
  (SSIMULACRA2 is RGB-only; raw alpha yields garbage/negative scores). Both candidate
  and reference are flattened identically, so visible content is compared fairly — but
  alpha-channel fidelity itself is *not* scored. Corroborate transparent fixtures via
  the `--visual` contact sheet.
- **Animated fixtures are scored at 84px only.** The pure-Python SSIMULACRA2 costs
  ~0.6 s/frame at 84px but ~6 s/frame at 320px, so a many-frame fixture at 320px makes
  the run intractable. File **size** at larger sizes is the size axis (no metric needed)
  and can be checked directly with `vipsthumbnail`/`gif2webp`. A compiled SSIMULACRA2
  binary would lift this limit (see `Ssimulacra2::$bin`).

## Corpus tiers (representative vs stress)

Each `manifest.json` entry has a `tier`, and the two are gated differently so "is Thumbro
better?" is never conflated with "does Thumbro stay safe?":

- **`representative`** — real, committed, freely-licensed images that mirror what wikis
  actually thumbnail (see `corpus/CREDITS.md` / `corpus/PROVENANCE.md`). These drive the
  full dominance **go/no-go**, and the printed `Summary` line counts only this tier.
  Representative static raster fixtures are measured at the MediaWiki 1.46 standardised
  content widths **180 / 250 / 400** (250 is the default unsized thumb); animated and
  stress fixtures keep their own targets.
- **`stress`** — synthetic pathologies and size extremes generated by
  `bin/make_corpus.php` (transparent animation, huge JPEG, over-threshold animation,
  tiny image). These are checked **only against the hard safety caps** — **PASS** (under
  every cap) or **CAP-BREACH** — and can never be a win or a loss.

Sourcing/licensing policy for the representative tier: CC0 / public-domain / CC-BY
(decision pending on widening to CC BY-SA — see `corpus/PROVENANCE.md`); photos from
Wikimedia Commons, sprites from CC0 game-asset libraries, gaming screenshots from
open-source games. Entries still awaiting a real licence-verified image are marked
`"pending_license": true` and held as synthetic stand-ins.

## Acceptance gate (summary)

The **representative** tier gates on **dominance vs ImageMagick** (the typical configured
MediaWiki scaler): PASS when the candidate is smaller at no-worse quality, or higher
quality at no-larger size; FAIL when ImageMagick dominates or a hard cap is breached
(quality < 50, time > 3 s static / 10 s animated, RSS > 512 MB); a genuine trade-off is
INCOMPARABLE (record a decision). A SSIMULACRA2 gap within
`GateThresholds::qualityWithinOfBaseline` (default 3) counts as a tie, so metric jitter
cannot decide a verdict.

**GD is a never-regress floor, not a win target.** GD is MediaWiki's *literal* default
scaler (`$wgUseImageMagick = false`); it is shown as `GD (floor)` and only flags a
**FLOOR BREACH** if GD dominates the candidate (which, being the lower-quality scaler, it
effectively never does). It is never counted as a win/loss.

**Performance is a safety cap, not a dominance axis.** Time and RSS never enter the
size/quality dominance comparison (generation is paid once and cached); they are bounded as
hard caps (an outright FAIL above the ceiling, so generation cannot choke the server) and
soft budgets (a material regression versus the baseline is flagged).

The **stress** tier uses a caps-only check (`AcceptanceGate::evaluateCaps`): PASS when the
candidate is under the hard **time and RSS** caps, CAP-BREACH otherwise. No baseline
comparison. **Quality is advisory here, not a hard cap** — stress fixtures are synthetic
pathologies (never served to users) and SSIMULACRA2 is unreliable on them (tiny / animated /
transparent), so a sub-floor score is flagged, not failed.

See `docs/adr/0001-image-benchmark-gate.md` for the rationale.

## Add a fixture
- **Stress fixture:** edit `bin/make_corpus.php`, run it, add a `"tier": "stress"` entry
  to `corpus/manifest.json`.
- **Representative fixture:** commit a real, freely-licensed image, record it in
  `corpus/CREDITS.md` + `corpus/PROVENANCE.md`, and add a `"tier": "representative"`
  entry with `license`/`source`/`author`. Do **not** add representative fixtures to
  `make_corpus.php` (they are curated, not generated).

## Add a contender
Implement `src/Contender.php` under `src/Contenders/`, register it in
`Orchestrator::__construct()` (as a baseline or candidate).
